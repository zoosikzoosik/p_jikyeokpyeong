직무역량평가 학습 앱 — 기술 스택 학습 문서

예상 학습 시간: 10~15분 · AI가 구현하는 동안 읽어보세요 · 6개 핵심 기술 다룸

이번 앱에서 등장하는 핵심 개념

1. Streamlit — pages/ 구조와 session_state

Streamlit은 Python 파일 하나가 웹 페이지 하나가 됩니다. pages/ 폴더에 파일을 넣으면 자동으로 다중 페이지 앱이 됩니다. 파일명 앞의 숫자(1_, 2_...)가 사이드바 메뉴 순서를 결정합니다.

문제 풀기 → 정답 제출 흐름에서 "이 사람이 로그인한 사용자인지"를 기억해야 합니다. 그 역할을 st.session_state가 합니다. Python 딕셔너리처럼 쓰면 됩니다.

# app.py — 로그인 성공 후
import streamlit as st

# 세션에 사용자 ID 저장
st.session_state["user_id"] = user["id"]

# pages/1_quiz.py — 다른 페이지에서 꺼내 쓰기
if "user_id" not in st.session_state:
    st.error("로그인이 필요합니다")
    st.stop()

user_id = st.session_state["user_id"]

API 키 같은 비밀값은 .streamlit/secrets.toml에 넣고, 코드에서 st.secrets["SUPABASE_URL"]로 읽습니다. 이 파일은 절대 Git에 올리면 안 됩니다.

⚠️ Streamlit은 버튼을 누를 때마다 파이썬 스크립트 전체를 위에서 아래로 다시 실행합니다. session_state에 저장하지 않은 변수는 매번 초기화됩니다.

2. Supabase Python 클라이언트 — 메서드 체이닝

Supabase는 PostgreSQL 데이터베이스를 REST API로 감싼 서비스입니다. Python 클라이언트는 SQL 대신 메서드를 이어 붙여서 쿼리를 만듭니다.

from supabase import create_client

supabase = create_client(url, key)

# SELECT * FROM users WHERE nickname = '테스트유저'
result = (supabase
    .table("users")
    .select("*")
    .eq("nickname", "테스트유저")
    .execute())

rows = result.data  # 항상 리스트
if rows:
    user = rows[0]
⚠️ .execute()를 빠뜨리면 쿼리가 실행되지 않습니다. 반환값이 빈 리스트가 아니라 쿼리 빌더 객체이므로 버그를 찾기 어렵습니다.

3. bcrypt — 비밀번호를 안전하게 저장하는 방법

비밀번호를 데이터베이스에 그대로 저장하면 DB가 털렸을 때 모든 비밀번호가 노출됩니다. bcrypt는 비밀번호를 알아볼 수 없는 해시로 바꿔서 저장합니다.

import bcrypt

# 회원가입: 비밀번호 → 해시
password = "mypassword"
hashed = bcrypt.hashpw(password.encode(), bcrypt.gensalt())
# 예: b'$2b$12$abc...' — 이것만 DB에 저장

# 로그인: 입력값과 해시 비교
is_ok = bcrypt.checkpw(password.encode(), hashed)
# True면 맞는 비밀번호
⚠️ bcrypt는 의도적으로 느리게 설계되어 있습니다. 로그인마다 약 0.1~0.3초 걸리는 것은 정상입니다. 이것이 보안의 핵심입니다.

4. Claude Vision API — 이미지에서 텍스트 추출

스캔 이미지를 Claude에게 보내면 텍스트를 읽어줍니다. 이미지는 base64로 인코딩해서 API 요청에 담습니다.

import base64
import anthropic

client = anthropic.Anthropic(api_key="...")

with open("scan.jpg", "rb") as f:
    image_data = base64.standard_b64encode(f.read()).decode()

response = client.messages.create(
    model="claude-sonnet-4-6",
    max_tokens=4096,
    messages=[{
        "role": "user",
        "content": [
            {"type": "image", "source": {
                "type": "base64",
                "media_type": "image/jpeg",
                "data": image_data
            }},
            {"type": "text", "text": "이 이미지의 텍스트를 모두 추출해주세요."}
        ]
    }]
)
extracted_text = response.content[0].text
⚠️ API 비용이 이미지 크기에 따라 달라집니다. 고해상도 스캔 이미지는 한 장당 토큰을 많이 씁니다.

5. Claude API로 문제 자동 생성 — JSON 응답 받기

Claude에게 "JSON 형식으로 답해주세요"라고 프롬프트를 짜면 구조화된 데이터를 받을 수 있습니다.

response = client.messages.create(
    model="claude-sonnet-4-6",
    max_tokens=8192,
    messages=[{
        "role": "user",
        "content": """다음 텍스트로 OX 문제 10개를 만들어주세요.
반드시 아래 JSON 형식으로만 응답하세요:
{"ox": [{"content": "...", "answer": "O", "explanation": "..."}]}
텍스트: {text}"""
    }]
)
import json
questions = json.loads(response.content[0].text)
⚠️ Claude가 항상 순수 JSON만 반환하지는 않습니다. 설명 문장을 앞뒤에 붙이는 경우가 있습니다. 응답에서 JSON 블록만 추출하는 파싱 방어 코드가 필요합니다.

6. pytest MagicMock — 외부 서비스 없이 테스트하기

테스트할 때마다 실제 Supabase나 Claude API를 호출하면 느리고 돈이 듭니다. MagicMock은 실제 객체처럼 행동하는 가짜를 만들어줍니다.

from unittest.mock import MagicMock

# 실제 supabase 대신 가짜 객체
mock_supabase = MagicMock()

# 특정 호출의 반환값 지정
mock_supabase.table.return_value \
    .select.return_value \
    .eq.return_value \
    .execute.return_value = MagicMock(data=[{"id": "abc"}])

# 테스트 대상 함수에 mock 주입
result = login("유저", "pass", mock_supabase)
⚠️ MagicMock의 체이닝은 처음에 헷갈립니다. .table().select().eq().execute()처럼 4단계 체인이면 return_value도 4번 이어줘야 합니다.

더 깊이 공부할 거리

  • Streamlit rerun model — 버튼 누를 때 스크립트 전체가 재실행되는 원리
  • bcrypt work factor — gensalt()의 숫자(기본값 12)가 의미하는 것
  • REST vs SDK — Supabase Python 클라이언트가 내부적으로 REST 요청을 만드는 방식
  • pytest fixtures (conftest.py) — 테스트 코드를 DRY하게 유지하는 방법

이번 구현의 핵심 개념 6가지

1. Streamlit 앱 구조 — pages/, session_state, secrets

이 앱은 app.py가 진입점이고 pages/ 아래 파일들이 각 화면입니다. Streamlit은 페이지 이동 시 스크립트를 새로 실행하므로, 인증 상태를 st.session_state에 유지해야 합니다. 실제 인증 흐름은 이렇습니다.

# components/auth.py
import streamlit as st

def is_authenticated() -> bool:
    return "user_id" in st.session_state

# pages/1_quiz.py 상단 — Guard 패턴
if not is_authenticated():
    st.warning("로그인이 필요합니다")
    st.switch_page("app.py")  # 진입점으로 리다이렉트
    st.stop()              # 이후 코드 실행 방지

secrets.toml은 로컬에서는 .streamlit/secrets.toml, Streamlit Cloud에서는 대시보드 설정으로 주입됩니다. 구조:

# .streamlit/secrets.toml
SUPABASE_URL = "https://xxx.supabase.co"
SUPABASE_KEY = "eyJ..."
ANTHROPIC_API_KEY = "sk-ant-..."

# 코드에서
url = st.secrets["SUPABASE_URL"]
⚠️ st.session_state는 브라우저 탭 단위입니다. 같은 URL을 새 탭으로 열면 세션이 분리됩니다. 또한 Streamlit 서버를 재배포하면 기존 세션이 날아가므로, 사용자가 재로그인해야 합니다.

관련 테스트 패턴

# tests/integration/test_auth.py
with patch("components.auth.st") as mock_st:
    mock_st.session_state = {"user_id": "some-uuid"}
    result = is_authenticated()
# st 모듈 자체를 mock하여 세션 상태를 제어한다

2. Supabase Python 클라이언트 — CRUD와 UPSERT

Supabase 클라이언트는 빌더 패턴(Fluent Interface)으로 쿼리를 구성합니다. 각 메서드가 자신을 반환하므로 체이닝이 가능합니다. 가장 복잡한 연산인 UPSERT 패턴은 user_progress 테이블에서 핵심적으로 쓰입니다.

# UPSERT: 없으면 INSERT, 있으면 UPDATE
supabase.table("user_progress").upsert(
    {
        "user_id": user_id,
        "question_id": question_id,
        "attempt_count": new_attempt_count,
        "correct_count": new_correct_count,
        "wrong_count": new_wrong_count,
        "last_correct": is_correct,
        "last_attempted_at": datetime.utcnow().isoformat(),
    },
    on_conflict="user_id,question_id"  # UNIQUE 제약 컬럼 명시
).execute()

spec에 정의된 UPSERT 로직: 기존 레코드를 SELECT 후 카운터를 Python에서 계산해 upsert합니다. DB 레벨에서 attempt_count + 1을 처리하는 방식이 아님에 주의합니다.

⚠️ UPSERT는 DB 레벨의 원자적 연산처럼 보이지만, 이 구현에서는 SELECT → 계산 → UPSERT를 Python에서 처리합니다. 동시에 같은 사용자가 같은 문제를 두 번 제출하면 카운터가 하나 누락될 수 있습니다(Read-Modify-Write race condition). 개인용 앱이라 실용적으로 무시 가능하지만, 프로덕션이라면 PostgreSQL의 ON CONFLICT DO UPDATE SET attempt_count = attempt_count + 1을 써야 합니다.

쿼리 체인 구조 읽는 법

# 테스트에서 mock 체인을 보면 실제 쿼리 구조를 역으로 알 수 있다
mock_supabase.table.return_value      # .table("users")
  .select.return_value                # .select("*")
  .eq.return_value                    # .eq("nickname", "...")
  .execute.return_value = MagicMock(data=[user])  # .execute()

3. bcrypt 비밀번호 해싱 — 저장과 검증

bcrypt는 salt(무작위 값)를 해시에 포함하므로 같은 비밀번호도 매번 다른 해시가 생성됩니다. checkpw는 해시에서 salt를 꺼내 비교하므로 별도로 salt를 저장할 필요가 없습니다.

# 회원가입 — 해시 생성
import bcrypt

def register(nickname: str, password: str, supabase) -> dict:
    if not nickname or not password:
        raise ValueError("닉네임과 비밀번호를 입력하세요")
    # 중복 확인 후...
    password_hash = bcrypt.hashpw(
        password.encode("utf-8"),
        bcrypt.gensalt()   # rounds=12 기본값
    ).decode("utf-8")    # DB에는 str로 저장

    result = supabase.table("users").insert(
        [{"nickname": nickname, "password_hash": password_hash}]
    ).execute()
    return result.data[0]

# 로그인 — 해시 검증
def login(nickname: str, password: str, supabase) -> dict | None:
    if not nickname or not password:
        return None
    result = supabase.table("users").select("*").eq("nickname", nickname).execute()
    if not result.data:
        return None
    user = result.data[0]
    ok = bcrypt.checkpw(password.encode("utf-8"), user["password_hash"].encode("utf-8"))
    return user if ok else None
⚠️ Timing attack: if not result.data: return None는 "닉네임 없음"과 "비밀번호 틀림"을 응답 시간으로 구분할 수 있게 합니다. 개인용 앱에서는 무시 가능하나, checkpw를 항상 실행하는 형태(더미 해시 비교)가 더 안전합니다.

4. Claude Vision API — 이미지 OCR 파이프라인

pipeline/ocr_and_generate.py에서 이미지를 base64로 인코딩해 Claude에 전달하고, 응답 텍스트를 바로 문제 생성 프롬프트에 재사용합니다. 핵심은 API 호출 2회가 순서대로 발생한다는 것입니다.

def ocr_image(image_path: str, client) -> str:
    with open(image_path, "rb") as f:
        raw = f.read()
    media_type = "image/jpeg" if image_path.endswith(".jpg") else "image/png"
    encoded = base64.standard_b64encode(raw).decode()

    response = client.messages.create(
        model="claude-sonnet-4-6",
        max_tokens=4096,
        messages=[{"role": "user", "content": [
            {"type": "image", "source": {
                "type": "base64", "media_type": media_type, "data": encoded
            }},
            {"type": "text", "text": PROMPTS["ocr"]}
        ]}]
    )
    return response.content[0].text

오류 처리 방식: spec에 따라 이미지별 오류는 건너뛰고 계속 진행합니다.

skipped = []
for img_path in image_paths:
    try:
        text = ocr_image(img_path, client)
        questions = generate_questions(text, client, count)
        # INSERT...
    except Exception as e:
        skipped.append(img_path)  # 실패 목록에만 기록, 중단 안 함
⚠️ base64 인코딩은 원본보다 약 33% 더 큰 문자열을 만듭니다. 고해상도 스캔 이미지(5MB)를 여러 장 처리하면 메모리 사용량이 급격히 늘어납니다. 배치 크기를 제한하거나 이미지를 리사이즈하는 전처리가 실제 운용에 필요합니다.

5. Claude API 구조화 출력 — JSON 문제 생성

Claude에게 JSON을 반환하게 하려면 프롬프트에 정확한 스키마를 제시해야 합니다. pipeline/prompts.py에 프롬프트를 분리하는 것은 LLM 출력 형식이 자주 조정되기 때문입니다.

# pipeline/prompts.py 구조 (예상)
QUESTION_PROMPT = """다음 교재 텍스트를 바탕으로 문제를 생성하세요.
반드시 아래 JSON 스키마를 정확히 따르세요. 설명 없이 JSON만 응답하세요.

스키마:
{{
  "ox": [{{"content": str, "answer": "O"|"X", "explanation": str}}],
  "fill": [{{"content": str, "answer": str, "explanation": str}}],
  "mcq": [{{"content": str, "answer": "1"|"2"|"3"|"4",
            "choices": [str, str, str, str], "explanation": str}}]
}}

OX {count}개, 빈칸 {count}개, 객관식 {count}개.
교재 텍스트:
{text}"""

파싱 방어 코드가 중요합니다. Claude가 마크다운 코드 블록으로 감싸는 경우가 있습니다.

import json, re

def parse_questions(raw: str) -> dict:
    # ```json ... ``` 블록 제거
    cleaned = re.sub(r"```(?:json)?\s*|\s*```", "", raw).strip()
    try:
        return json.loads(cleaned)
    except json.JSONDecodeError:
        # JSON 블록만 추출 시도
        match = re.search(r"\{.*\}", cleaned, re.DOTALL)
        if match:
            return json.loads(match.group())
        raise
⚠️ Claude가 생성한 JSON의 "answer" 필드가 "1" 대신 1(정수)로 올 수 있습니다. DB 스키마에서 answer는 text 타입이므로, Supabase INSERT 전에 문자열 변환을 거쳐야 합니다. 테스트 mock에서는 이 케이스가 잡히지 않을 수 있습니다.

6. pytest mock 패턴 — MagicMock, patch, stateful mock

이 프로젝트의 테스트는 3가지 mock 패턴을 사용합니다. 각각 다른 상황에 적합합니다.

패턴 1: return_value 체인 설정 — 단일 쿼리 결과를 지정할 때

# test_auth.py 패턴
mock_supabase.table.return_value.select.return_value\
    .eq.return_value.execute.return_value = MagicMock(data=[sample_user])

# 장점: 간단하고 직관적
# 단점: 같은 메서드를 다른 인자로 두 번 호출하면 구분 불가

패턴 2: side_effect 함수 — 호출 인자에 따라 다른 결과를 반환하거나 상태를 캡처할 때

# test_auth.py — insert 호출 내용 캡처
inserted = []
def capture_insert(data):
    row = data[0] if isinstance(data, list) else data
    inserted.append(row)
    m = MagicMock()
    m.execute.return_value = MagicMock(data=[row])
    return m

mock_supabase.table.return_value.insert.side_effect = capture_insert
# 테스트 후 inserted[0]["password_hash"]를 검증

패턴 3: stateful mock — 여러 번 호출 사이에 상태가 누적되어야 할 때

# test_user_progress.py — _make_stateful_progress_mock
# progress_store dict를 클로저로 공유하여
# upsert 호출이 state를 변경하고
# select 호출이 변경된 state를 반환한다
progress_store = {}

def upsert_chain(data, **kwargs):
    row = data[0] if isinstance(data, list) else data
    key = (row["user_id"], row["question_id"])
    progress_store[key] = row  # 상태 갱신
    u = MagicMock()
    u.execute.return_value = MagicMock(data=[row])
    return u

# 이로써 3회 upsert_progress 호출 후
# progress_store에서 최종 상태를 검증 가능

패턴 4: side_effect 리스트 — 순서대로 다른 결과를 반환해야 할 때

# test_pipeline.py — OCR 후 문제 생성 순서
mock_client.messages.create.side_effect = [
    vision_response,    # 1회차: OCR 응답
    generate_response,  # 2회차: 문제 생성 응답
]
# 3회 호출하면 StopIteration 발생 → 의도치 않은 추가 호출 방어
⚠️ side_effect에 리스트를 넣으면 순서대로 소진됩니다. 예상보다 많이 호출되면 StopIteration이 발생합니다. 이것은 버그처럼 보이지만 실은 "예상 외 API 호출이 있다"는 것을 알려주는 테스트 안전장치입니다.

아키텍처 결정과 장기적 관점

1. Streamlit의 구조적 한계와 이 선택의 의미

Streamlit은 "상태가 없는 함수형 UI"와 "상태를 가진 앱" 사이 어딘가에 있습니다. 모든 인터랙션이 전체 스크립트 재실행을 트리거하므로, 복잡한 상태를 session_state에 담을수록 재실행 비용이 늘어납니다. 이 앱에서 문제 리스트와 현재 인덱스를 session_state에 저장하는 구조는 문제 수가 수백 개가 되어도 영향이 없지만, 사용자가 수십 명으로 늘어나면 Streamlit Cloud의 메모리 제한에 부딪힙니다.

장기적으로 이 앱이 팀용으로 확장된다면, 현재 architecture는 교체 비용이 큽니다. 그것을 알면서도 Streamlit을 선택한 것은 "개인용 + 빠른 시작"이라는 맥락에서 옳은 결정입니다. 단, 이 결정을 한 달 뒤 잊으면 위험합니다.

2. UPSERT 설계와 동시성 — 지금은 괜찮지만

현재 구현은 SELECT → 계산 → UPSERT 패턴입니다. 개인용 앱에서 한 사람이 동시에 두 탭에서 같은 문제를 제출할 가능성은 낮지만 zero는 아닙니다. 모바일 탭 + 데스크탑 탭 시나리오를 생각해보세요.

진짜 원자적 업데이트가 필요하다면:

-- Supabase SQL Editor에서 RPC 함수 정의
CREATE OR REPLACE FUNCTION upsert_progress(
    p_user_id uuid, p_question_id uuid, p_is_correct bool
) RETURNS user_progress AS $$
  INSERT INTO user_progress (user_id, question_id, attempt_count,
      correct_count, wrong_count, last_correct, last_attempted_at)
  VALUES (p_user_id, p_question_id, 1,
      (p_is_correct::int), ((NOT p_is_correct)::int),
      p_is_correct, now())
  ON CONFLICT (user_id, question_id) DO UPDATE SET
      attempt_count = user_progress.attempt_count + 1,
      correct_count = user_progress.correct_count + (p_is_correct::int),
      wrong_count = user_progress.wrong_count + ((NOT p_is_correct)::int),
      last_correct = p_is_correct,
      last_attempted_at = now()
  RETURNING *;
$$ LANGUAGE sql;

# Python에서
supabase.rpc("upsert_progress", {
    "p_user_id": user_id,
    "p_question_id": question_id,
    "p_is_correct": is_correct
}).execute()

이 방식은 SELECT 왕복이 없어 레이턴시도 반으로 줄어듭니다. 현재 테스트 구조도 stateful mock으로 되어 있어 RPC 방식으로 전환해도 인터페이스 변경만으로 테스트 재작성 없이 가능합니다.

3. OCR 파이프라인의 멱등성 설계 — Guard 패턴

spec에 "동일 파트명이 이미 존재하면 오류 출력 후 중단"이 있습니다. 이것은 멱등성 설계 중 단순한 형태인 Guard입니다. 재실행해도 안전한(at-most-once) 오퍼레이션으로 만든 것입니다.

이 설계의 trade-off: 스크립트 중간에 실패했을 때(이미지 10개 중 5개 처리 후 크래시) 재시작이 불가능합니다. 지금은 파트 단위 전체 재처리(수동 삭제 후 재실행)만 지원합니다. 더 정교한 설계라면 이미지 단위 체크포인트를 두겠지만, 1회성 작업에 과설계입니다.

Guard가 SystemExit, ValueError, RuntimeError 중 무엇을 raise하는지가 아직 미결정입니다. 테스트에서 세 가지를 모두 허용(pytest.raises((SystemExit, ValueError, RuntimeError)))하고 있어, 구현자가 어떤 것을 선택해도 통과합니다. 팀 프로젝트라면 하나로 통일해야 할 결정입니다.

4. 테스트 설계 — 구현 디테일이 아닌 행위를 검증하는가

이 프로젝트의 테스트 품질 포인트:

잘 된 점: AC4(3회 풀어 2회 정답) 시나리오를 stateful mock으로 검증합니다. 구현이 "SELECT 후 계산"이든 "RPC"든 외부 인터페이스(upsert_progress 함수 시그니처)만 맞으면 테스트가 통과합니다. 이것이 진짜 행위 기반 테스트입니다.

주의할 점: test_register_stores_bcrypt_hash_not_plaintext는 구현 디테일(insert 호출 내용)을 캡처합니다. 구현이 "insert 대신 upsert를 쓰도록" 바뀌면 테스트가 깨집니다. 이 테스트가 검증하는 것은 "비밀번호가 평문이 아닌 해시로 저장된다"이므로, 가능하면 실제 DB에 조회하는 방식이 더 강건하지만, mock 환경에서의 타협입니다.

내가 리뷰어라면 던질 질문 5가지

secrets.toml이 .gitignore에 등록되어 있는가? Streamlit Cloud에서는 환경변수로 주입하는데, 로컬에서 실수로 commit하는 사고가 자주 발생한다.
Claude Vision 호출 시 이미지 크기 제한을 두고 있는가? 5MB 이상의 JPEG는 base64 후 ~7MB 문자열이 된다. Anthropic API의 이미지 크기 제한(20MB)과 token limit 모두 고려해야 한다.
Supabase anon key와 service_role key를 구분하고 있는가? anon key는 RLS 정책을 따르지만 service_role key는 RLS를 우회한다. 이 앱에서 RLS를 설정하지 않았다면 anon key로도 모든 데이터에 접근 가능하다. 개인용이라면 허용 범위지만 인식은 해야 한다.
문제 생성 파이프라인에서 Claude가 count보다 적은 문제를 반환할 때 어떻게 처리하는가? "OX 10개 생성"을 요청했는데 7개만 반환하면 그냥 7개를 INSERT하는가, 재시도하는가?
is_authenticated()가 st.session_state["user_id"] 존재 여부만 확인한다. user_id가 Supabase에 실제로 존재하는지는 확인하지 않는다. 누군가 개발자 도구로 session_state를 조작할 수 있는가? Streamlit에서는 불가능하지만, 세션 하이재킹 벡터를 이해하고 있는지 묻고 싶다.

운영 관점 — 장애 시나리오

예상 가능한 장애 유형

  • Supabase 무료 플랜 일시 중지: 비활성 프로젝트는 1주 후 pause됩니다. 로그인 자체가 불가능해지므로 주기적으로 접속하거나 유료 플랜으로 전환 필요
  • Streamlit Cloud 메모리 초과: 무료 플랜은 앱당 1GB 메모리. 문제 수가 1000개를 넘으면 전체 데이터 로딩 방식 재검토 필요
  • Claude API rate limit: OCR 파이프라인에서 이미지를 빠르게 처리하다 rate limit에 걸리면 Exception이 발생하고 이미지는 skipped_images에 쌓입니다. 배치 처리 시 이미지 사이에 sleep을 추가하거나 retry backoff 로직 필요
  • user_progress ON CONFLICT 실패: UNIQUE(user_id, question_id)가 DB에 없으면 upsert가 매번 INSERT를 시도해 중복 레코드가 생깁니다. 마이그레이션 확인 필수
